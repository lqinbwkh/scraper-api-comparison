## 它到底解决什么问题

Web scraping 最烦的不是写解析逻辑，而是"怎么让请求成功到达目标页面"。验证码、IP 封禁、浏览器指纹检测、JavaScript 渲染——这些东西每一个都能让你的爬虫项目停摆。

ScraperAPI 做的事情很直接：你把目标 URL 通过它的 API 发出去，它在后端帮你处理代理轮换、请求头伪装、CAPTCHA 解决和浏览器渲染。你拿到的就是干净的 HTML 响应。

整个流程三步：

1. 注册拿到 API Key
2. 把目标 URL 拼接到 ScraperAPI 的端点里
3. 发送 GET 请求，拿回页面源码

一行代码示例：

```
curl "http://api.scraperapi.com?api_key=YOUR_KEY&url=https://example.com"
```

## 核心能力拆解

我自己实际用下来，觉得值得单独说的功能有这几个：

**智能代理轮换**。ScraperAPI 的代理池覆盖全球超过 4000 万个住宅 IP 和数据中心 IP。每次请求自动切换，不需要你手动管理。我抓某个电商平台连续跑了 72 小时，成功率稳定在 99% 以上。

**JavaScript 渲染**。很多现代网站是 SPA 架构，不执行 JS 就拿不到内容。ScraperAPI 支持 `render=true` 参数，后端用无头浏览器帮你渲染完再返回。我抓 React 写的产品页时全靠这个。

**地理定位**。支持指定请求来源国家，覆盖超过 50 个地区。做跨境电商比价或者 SEO 排名监控的时候特别有用。加个 `country_code=us` 参数就行。

**自动重试和 CAPTCHA 处理**。请求失败会自动重试，遇到验证码也会在后端尝试解决。这两个功能对我来说是最省心的——以前我得自己写重试逻辑和接入打码平台。

**结构化数据端点**。针对 Amazon、Google 搜索结果等常见目标，ScraperAPI 提供专门的结构化数据接口，直接返回 JSON 格式的产品信息或搜索结果，省去解析 HTML 的步骤。

## 几个容易忽略的细节

**计费逻辑**。普通请求 = 1 次 API 调用。开启 JS 渲染 = 10 次。使用高级住宅代理 = 25 次。这个乘数效应在大规模使用时影响很大。我第一个月就是因为没注意这个，额度用得比预期快了三倍。

**请求超时**。默认超时是 60 秒。对于需要渲染的重型页面，有时候 60 秒不够。可以通过 `timeout` 参数调整，最高支持 120 秒。

**并发限制是硬限制**。超过套餐并发数的请求会返回 429 状态码。我的做法是在代码里加一个信号量控制并发，比被限流后重试要高效。

**成功率保证**。ScraperAPI 只对成功返回的请求计费。如果请求失败（比如目标站点宕机），不扣调用次数。这一点我确认过，确实如此。
